3개월간 점진적으로 진화시킨 V2X 보안 통신 데몬(v2x-httpsd)

#C++ 리눅스 데몬 개발 #컴파일 타임 플러그인 아키텍처 #한국-베트남 글로벌 협업

OEM(완성차 기업)별 전용 통신 데몬이 개발되기 전까지 V2X 공정을 진행할 수 있는 범용 테스트 데몬으로,
한국과 베트남 2개국 팀이 협업하여 IPC-HTTPS Bridge Framework를 개발했습니다.

기간: 2025.03 ~ 2025.06 | 역할: v2x-httpsd 아키텍처 설계 및 전체 코드 구현

참여 구성원
  • • 본인: v2x-httpsd 아키텍처 설계, 전체 코드 구현
  • • LG전자 V2X Manager팀: 요구사항 정의 및 확인
  • • LG전자 베트남 지사: v2x manager 개발, v2x-httpsd 연동
💡 프로젝트 배경

완성차 기업(OEM)별로 통신 데몬이 달라 공정 계획 지연이 발생하는 상황에서, OEM이 자체 데몬을 제공하기 전까지 사용할 수 있는 범용 테스트 데몬을 개발하여 공정 계획 수립을 가능하게 했습니다. 즉, LG전자 내부에서 다양한 프로젝트에 즉시 활용할 수 있는 솔루션으로 개발되었습니다.

컴파일 타임 플러그인 아키텍처 설계

핵심 설계 철학

JLR, BMW 등 각기 다른 OEM의 요구사항을 모두 충족시키기 위해 OCP(개방-폐쇄 원칙)와 템플릿 특수화를 '정적 플러그인'처럼 활용했습니다. 이를 통해 새로운 기능이 추가되어도 기존 코드를 수정할 필요가 없는 안정적인 구조를 구현했습니다.

⚡ 계약에 의한 설계: 템플릿 프레임워크의 오용을 방지하기 위해 2단계 안전장치(컴파일 타임 체크 + 런타임 에러 메시지)를 구현하여 개발자 경험을 향상시켰습니다.

❔ 독자 규격의 쉐어드 메모리 기반 IPC: LG전자에서 개발한 쉐어드 메모리 방식의 IPC

🌱 Spring Framework 철학을 C++에 적용

SSAFY에서 학습한 Spring의 DI/IoC 개념을 C++에 창의적으로 적용하여, IPC_API<T> 인터페이스로 구현체 교체가 가능한 구조를 설계했습니다. 계층 분리와 템플릿 메서드 패턴으로 확장성을 확보한 것이 특징입니다.

▼
상세 구현 내용

v2x-httpsd 개발은 C++로 진행되었지만, 그 구조와 설계 철학 전반에는 SSAFY 교육을 통해 학습한 Spring Framework의 개념이 깊숙이 반영되었습니다. C언어 스타일의 단순 구현을 넘어, 객체지향 원칙에 기반한 유연하고 확장 가능한 아키텍처를 구축하는 것이 목표였습니다.

1. 의존성 주입(IoC/DI)과 인터페이스를 통한 확장성

Spring을 통해 "Java 인터페이스를 활용하면 코드 변경 없이 기능을 확장할 수 있다"는 점에 크게 감명받았고, 이 개념을 C++ 프로젝트의 핵심 설계 원칙으로 삼았습니다.

  • IPC_API라는 추상 인터페이스를 설계하여, 독자 규격의 쉐어드 메모리 기반 IPC, POSIX Queue 등 다양한 IPC 구현체를 쉽게 교체 가능
  • run_daemon 함수는 Spring의 Controller처럼, 구체적인 구현체에 의존하지 않고 전체 흐름을 조율
2. 계층 분리(Layer Separation) 아키텍처

IPC 통신 로직을 두 개의 계층으로 분리: read/write는 비즈니스 로직을, read_impl/write_impl은 실제 I/O를 담당. 이는 템플릿 메서드 패턴의 형태가 되었습니다.

3. 로직 집중을 위한 추상화

BinaryStreamDataConverter<T>라는 공통 클래스로 모든 IPC 구현체가 동일한 직렬화/역직렬화 로직을 공유. 개발자는 핵심 기능 구현에만 집중할 수 있게 했습니다.

4. Hook 패턴과 안정적인 프로그램 종료

Spring의 인터셉터 개념을 차용하여, C++의 정적 변수 초기화 순서를 활용한 Graceful Termination Hook을 구현하여 시스템의 안정성을 높였습니다.

The Challenge

  • 불확실한 개발 일정: OEM별 전용 통신 데몬 개발 완료 전까지 V2X 공정을 진행할 수 있는 대체 솔루션 필요
  • 다양한 최종 고객사(OEM) 요구사항: JLR, BMW 등 여러 프로젝트에서 모두 사용 가능해야 함
  • 서로 다른 통신 방식(IPC): 독자 규격의 쉐어드 메모리 기반 IPC, POSIX Queue, Unix Domain Socket 등 각기 다른 IPC 지원 필요
  • 외부 라이브러리 제약: 고객사(LG전자)의 독자적인 IPC 라이브러리(독자 규격의 쉐어드 메모리 기반 IPC)는 기능적 한계가 명확
  • 복잡한 개발 환경: Yocto Linux 기반의 복잡한 빌드 및 디버깅 환경

My Approach & Key Decisions

1. 확장성을 고려한 아키텍처 설계

문제: 새로운 고객사나 SDK가 추가될 때마다 전체 코드를 수정해야 하는 리스크
해결: 템플릿 특수화와 OCP(개방-폐쇄 원칙)를 적용하여, run_daemon이라는 핵심 로직은 전혀 수정하지 않고, 각 데이터 타입과 IPC 방식에 대한 구현체 파일만 추가하면 새로운 요구사항을 즉시 지원할 수 있는 구조 완성

2. 라이브러리(독자 규격의 쉐어드 메모리 기반 IPC)의 한계를 디자인 패턴으로 극복

문제: '독자 규격의 쉐어드 메모리 기반 IPC'는 원하는 토픽의 메시지만 선택적으로 읽을 수 없는 치명적 제약
해결: Producer-Consumer 패턴을 적용한 독자 규격의 쉐어드 메모리 기반 IPC Monitor 클래스 구현

  • 백그라운드 스레드: 모든 메시지를 수신하여 토픽별 큐에 자동 분류
  • 메인 스레드: 일반적인 동기식 I/O처럼 원하는 토픽의 메시지만 간편하게 조회

3. 견고한 시스템 안정성 설계

• NTPClient 개발: 요구사항에는 없었지만, 다른 OEM 프로젝트 경험상 시스템 시간 오류가 인증서 문제를 일으킬 것을 예측하여 개발 (2주 허비 회피)
• Graceful Shutdown: SIGSEGV 같은 Fatal Signal 발생 시에도 자원 누수를 방지하는 안전한 종료 메커니즘 구현
• 안전한 로깅: HTTP 요청/응답의 바이너리 값 출력으로 인한 터미널 깨짐 방지를 위해 Content-Type 기반 출력 변환 구현
• 디버깅 효율: SRP(단일책임 원칙) 적용으로 문제 원인 파악 시간을 16배 단축 (1일 → 30분)

점진적 진화 전략: 실패에서 배운 리스크 관리

배경: 과거 ASN.1 자동화 프로젝트에서 한 달간 결과물 없이 팀 업무를 지연시킨 뼈아픈 실패
→ "제때 동작하는 결과물을 못 내면, 전체 프로젝트가 멈춘다"는 교훈

위기관리 전략:
• Week 1-3: C 스타일 프로토타입으로 3주 내 동작하는 버전 제공

→ 동작은 유지하면서 코드 품질 개선
• Week 6-9: Spring 철학 적용으로 미래 확장성 확보
→ BMW 프로젝트 추가 시 즉시 대응 가능한 구조 완성

결과: 고객 일정은 지연시키지 않으면서, 최종적으로는 견고한 아키텍처 달성

#네트워크 난제 해결

선임 개발자들이 해결하지 못한 문제를 2주 만에 해결하여 6개월 이상의 공정 계획 선행 기여

기간: 2025.02 | 역할: 문제 원인 분석 및 해결 방안 제시/구현

The Challenge

  • 3개월간의 해결 시도 실패 상태: LTE 미지원 상태의 텔레매틱스 시료가 외부 인터넷 서버와 통신 불가
  • LG전자 사내 보안 프로그램으로 인한 PC 기능(ICS) 사용 불가
  • 시료 내부의 복잡한 네트워크 구조 (듀얼 리눅스, 브릿지, 방화벽)
  • LG 인프라팀의 비협조적인 상황

My Approach & Action

1. 근본 원인 분석:
기존의 실패한 시도를 반복하는 대신, 문제의 본질을 파고들었습니다. LG전자 내부 Confluence와 코드비머(OEM과 약속된 공정계획 추적 웹페이지)의 방대한 아키텍처 문서를 분석하여, 문제가 단순한 PC 설정이 아닌 '시료의 구조적 제약'에 있음을 명확히 규명했습니다.
• '어플리케이션 리눅스'는 '네트워크 리눅스'를 통해서만 외부와 통신하는 프록시 구조
• 또한 보안을 위해 필요한 포트 외 모든 네트워크 차단, Default Gateway 설정 미동작

2. 창의적인 해결책 제시 및 구현:
하드웨어 추가 없이, PC의 소프트웨어 기능을 활용하는 새로운 해결책을 고안했습니다.
• 과거 국방부 데이터센터에서 다뤘던 스위치 장비의 NAT 기능을 떠올리고, 이를 PC 환경에서 소프트웨어적으로 구현할 방법을 모색
• Hyper-V 가상 스위치를 이용하여 PC 자체를 라우터처럼 동작시키는 방법을 고안하고, 시료의 방화벽 및 라우팅 테이블을 직접 설정하여 인터넷 연결에 성공

3. 지식의 자산화 및 공유:
혼자만의 해결책으로 남기지 않았습니다. 해결 과정을 상세한 매뉴얼로 만들어 팀 전체에 공유하여, 누구나 5분 안에 동일한 개발 환경을 구축할 수 있도록 표준화했습니다.

The Owner

경계를 넘나드는 오너십

#기술 컨설팅 #선제적 문제 해결

프로젝트의 성공을 위해 스스로 문제를 찾아내고 경계 없이 해결하는 '전략적 파트너'

기간: 2024.11 ~ 2025.07 | 역할: V2X 보안 기술 컨설턴트, 문제 해결사

Story 1: 기술적 통찰력으로 6개월의 프로젝트 지연을 막다

상황: 고객사(LG전자)가 대기 시간을 효율적으로 쓰기 위해, 미리 IEEE 1609.2.1 표준을 기반으로 HTTP Body만으로 통신하는 계획을 수립.
최종 고객사(JLR)의 인증서 다운로드 계획이 확정되지 않은 상황에서, 기술 자문가로서 최종 통합 단계의 전면적인 공정 변경 리스크를 사전에 방지하는 역할을 수행 필요

나의 접근법:
• '지적'이 아닌 '설득': 일방적 오류 지적 대신, 고객사가 스스로 최적의 결정을 내리도록 기술 조력자로 접근
• 객관적 데이터 제시: 표준 문서의 REST API 명세를 시각화한 보고서로 기술적 근거 명확히 전달
• 눈높이 소통: 복잡한 표준 용어를 쉬운 언어로 풀어서 의사결정권자도 문제의 심각성을 즉시 이해

Story 2: 내/외부 장애물을 직접 해결하는 주도성

상황: v2x-httpsd 개발 중 3rd-party SDK 버그 및 누락된 파일, 내부 자사(새솔테크) 테스트 서버 제약으로 개발 진행 차질
상용화 단계에 있는 V2X Security 기술에 대한 자사의 PoC 경험, 나라별 표준 분석, 여러 업체와의 상호호환성 테스트, PKI Client 개발 등 제품 출시 단계에서의 경험을 활용하여, 내부 동작을 예측하고 문제를 해결할 수 있는 V2X 도메인 전문가로서 독립적인 문제 해결 능력을 발휘하는 역할을 수행 필요

나의 행동:
• 외부 문제 해결: SDK 버그 및 누락된 파일 발견 즉시 글로벌 이슈 트래킹 시스템에 직접 리포트 → 헝가리 3rd-party사로부터 10일 만에 패치 약속 확정
• 내부 문제 해결: 테스트 서버 제한 문제를 담당 부서와 직접 협의하여 해제 → 팀 전체 대기 시간 90% 감소
• 핵심 마인드: "팀의 성공이 곧 나의 성공이라는 마인드로 적극 협력"

Project 04

The Global Detective

기술적 증거로 글로벌 협업의 실마리를 풀다

#서버 이슈 분석 #글로벌 협업 #다국적 이해관계가 얽힌 문제

여러 국가(한국, 미국, 헝가리)의 이해관계자가 얽힌 복잡한 서버 이슈 상황에서,
명확한 기술적 증거를 바탕으로 문제의 핵심을 파고들어 해결의 실마리를 제공하고, 지연될 뻔했던 프로젝트를 재개했습니다.

기간: 2025.03 ~ 2025.04 | 역할: 글로벌 이슈 분석 및 해결 주도

The Challenge

  • 미궁에 빠진 이슈: 인증서 발급 과정에서 원인 불명의 서버 에러(`ec not exist`)가 발생
  • 복잡한 이해관계: 국내 고객사(LG전자), 해외 PKI 클라이언트사(Commsignia), 해외 서버사(GHS ISS) 등 여러 이해관계자가 얽혀있어 책임 소재가 불분명하고, 시차로 인해 커뮤니케이션이 지연되는 상황

My Approach & Action

1. 가설 기반의 논리적 추론

불명확한 에러 메시지(`ec not exist`)와 IEEE 표준에 대한 깊은 이해를 바탕으로, '서버가 Enrollment Certificate를 인식하지 못한다'는 핵심 가설을 수립했습니다.

2. 데이터 기반의 증거 확보

가설을 증명하기 위해, PKI 클라이언트가 생성한 인증서 요청 바이너리를 ASN.1 디코딩하여 직접 분석했습니다. 그 결과, 요청문에는 Enrollment Certificate가 정상적으로 포함되어 있음을 확인하며, 문제의 원인이 클라이언트가 아닌 서버 측에 있음을 명확히 증명했습니다.

3. 동료를 위한 선제적 대응

해외 서버의 잦은 일정 지연으로 이미 최종 고객사(JLR)로부터 에스컬레이션(Escalation) 압박을 받고 있는 상황이었습니다. 여기서 또다시 책임 공방으로 시간이 지체되면, 시차 문제 등으로 인해 결국 고객사 동료들이 과도한 부담을 떠안게 될 것을 알기에, 분석 과정을 상세히 기술한 객관적인 기술 보고서를 작성하여 국내 고객사(LG전자)에 전달했습니다. 이 보고서는 최종 고객사(JLR)가 명확한 증거를 가지고 해외 서버사에 문제 해결을 요청하는 강력한 '트리거' 역할을 했습니다.
